# Backend Management Product Rules

适用范围：管理后台页面的模块职责、页面内容语义、对象详情结构、工作台行为与运营产品规则。

## 一、分层边界

- 本文件只定义“后台页面要解决什么问题、承接什么对象、提供什么动作、支持什么跳转”。
- 后台 `.pen` 画布的布局、视觉、组件、列表形态、抽屉形态、图表用法与页面骨架，统一以下沉到 `docs/rules/backend-management-design-rules.md` 为准。
- 指标公式、异常阈值、SLA、权限边界、审计字段、导出留存等实现契约，统一以下沉到 `docs/rules/backend-development-rules.md` 为准。
- 如果页面结构与内容语义发生冲突，先校正本文件与正式 `.pen`，再进入后端实现细化。

## 二、后台 V1 信息架构与模块职责

当前后台 V1 固定按以下 7 个一级模块组织：

1. 运营总览
2. 商品管理
3. 内容配置
4. 社区审核
5. 订单履约
6. 用户 / 服务运营
7. 系统配置

模块职责约束：

- `用户 / 商家` 统一更名为 `用户 / 服务运营`，避免把当前平台主后台误画成商家自运营后台。
- `用户 / 服务运营` 下当前至少包含以下二级页面：
  - `用户 360 运营台`
  - `统一治理中心`
  - `医生 / 问诊运营`
- `运营总览` 负责宏观经营看板、跨域风险优先级与高频跳转入口，不承接全部明细处理。
- `商品管理` 负责商品列表、上下架、商品详情内容配置工作台与商品级经营表现下钻。
- `内容配置` 负责首页精选、内容位、排期、发布与前台落点配置，不和商品基础资料页混成一个页面。
- `社区审核` 负责社区内容审核、申诉回查、误杀纠偏与社区健康洞察。
- `订单履约` 负责履约时效、物流异常、售后超时与升级工单处理，不应退化为普通订单状态列表。
- `系统配置` 负责系统级模板、字典、角色、默认阈值、文案位、模块开关等基础能力配置。

## 四、宏观看板与对象处理闭环

- 每个模块都要同时有：
  - 宏观看板
  - 微观对象处理入口
- 宏观看板负责：
  - 看趋势
  - 看异常
  - 看结构
  - 看优先级
- 微观处理负责：
  - 进入具体对象
  - 看详情
  - 做动作
  - 看日志
  - 看时间线
  - 回到列表
- 总览页的异常、榜单和动作项必须能跳到对应工作台或对象详情。
- 工作台条目必须能进入对象详情。
- 轻量动作优先在当前工作台完成，复杂处理再跳转到专题页。
- 后台不能做成“只能看趋势，不能处理对象”的半成品，也不能退化成“只有表格，没有经营判断”的流水后台。

## 五、运营总览的产品语义

运营总览首页不是纯 BI 页，也不是纯待办页。它要同时解决“今天业务怎么样”和“今天先处理什么”。

首页内容结构约束：

- 顶部 4 个核心经营卡固定为：
  - `支付成交总额`
  - `内容带成交额`
  - `精选点击率`
  - `归因支付订单数`
- 第二排 4 个健康指标固定为：
  - `发货准时率`
  - `投诉率`
  - `超 SLA 单量`
  - `售后平均处理时长`
- 中部 3 个工作区固定为：
  - `今日优先处理`
  - `风险 / 异常诊断`
  - `热门商品榜`
- 下部 2 个分析区固定为：
  - `内容转化链主趋势图`
  - `最近关键动作`

首页细化约束：

- 首页榜单固定优先 `热门商品榜`。
- `热门商品榜` 的排序口径固定按 `内容带成交`，不是按浏览量或单纯销量。
- `热门医生榜` 不放在首页，应下沉到 `医生 / 问诊运营` 专题页。
- 用户复购率、复诊贡献等深专题指标不应挤占首页首屏，应放到对应专题工作台。
- 首页快捷动作固定优先承接：
  - `新建精选排期`
  - `处理高优投诉`
  - `审核高风险内容`
  - `查看履约异常`
- `最近关键动作` 只记录高风险或高影响动作，不记录普通浏览或低价值点击。
- 首页所有异常项、榜单项、快捷动作都必须有明确跳转目标，能够回到具体工作台、列表、对象详情或原版本记录。

## 六、商品管理的产品语义

商品管理当前至少拆成两个核心页面：

1. `商品列表 / 上下架管理`
2. `商品详情内容配置工作台`

执行约束：

- 商品列表页负责搜索、筛选、状态管理、上下架动作、经营表现概览与对象下钻。
- 商品详情页不是单纯商品资料页，而是“商品详情内容配置工作台”。
- 商品详情内容必须按结构化模块维护，不允许退化成一块富文本大字段。
- 商品详情内容工作台至少显式承接以下模块：
  - `首图 / 图集`
  - `核心卖点`
  - `功效说明`
  - `使用方式`
  - `规格参数`
  - `成分 / 材质`
  - `服务承诺`
  - `FAQ`
  - `相关推荐`
  - `评价摘要`
- 热门商品的宏观榜单可以放在首页，但商品经营分析的完整下钻要回到商品管理域内完成。

## 七、内容配置的产品语义

内容配置模块当前重点承接首页精选、内容位与前台动态配置，不应只理解为文案设置页。

执行约束：

- 首页精选、商品详情推荐位、活动内容位等，都应支持通过后台动态上架、下架、推送与替换。
- 内容配置对象必须显式承接以下状态与动作：
  - `草稿`
  - `预览`
  - `排期`
  - `发布`
  - `下线`
  - `版本`
  - `回滚`
- 预览必须同时展示：
  - 前台页面中的实际落点
  - 当前配置与前台模块的映射关系
- 排期能力默认是产品级能力，不允许依赖人工记时上线。
- 首页精选配置必须能看见候选池、当前在线内容、即将生效内容与历史版本。
- 涉及跳转的内容位，必须显式配置目标对象与跳转类型，不能只写一个模糊链接字段。

## 八、社区审核的产品语义

社区审核不是单纯审核列表，而是“审核工作台 + 内容健康洞察”组合页。

执行约束：

- 社区审核页必须同时承接：
  - `待审核 / 待复核 / 申诉回查` 等处理队列
  - `内容健康洞察`
- 社区健康洞察至少覆盖以下指标：
  - `发帖量`
  - `举报率`
  - `拦截率`
  - `申诉率`
  - `误杀纠偏率`
- 社区内容详情应能查看对象内容、举报原因、证据摘要、AI 判定结果、人工结论、处置历史与关联用户。
- 高风险社区个案如果进入跨域投诉或治理，应可跳入 `统一治理中心` 继续处理。

## 九、订单履约的产品语义

- 订单履约页默认按 `风险 + 时效` 工作台理解，不按简单订单状态列表理解。
- 页面重点应围绕：
  - `发货准时`
  - `物流异常`
  - `售后超时`
  - `升级治理`
- 履约异常条目必须能下钻到具体订单、售后单或异常事件详情。
- 履约页的优先级排序，默认围绕风险等级、SLA、用户影响范围，而不是单纯按创建时间倒序。

## 十、用户 / 服务运营的产品语义

### 用户 360 运营台

- 用户 360 默认按 `服务 / 风险优先` 组织，不按传统增长 CRM 优先组织。
- 用户详情至少覆盖以下区块：
  - `基础资料`
  - `宠物摘要`
  - `订单 / 售后历史`
  - `投诉 / 反馈 / 举报历史`
  - `问诊 / 医生服务历史`
  - `社区行为 / 风险标签`
  - `最近关键动作`

### 统一治理中心

- 统一治理中心按 `紧急度 / SLA` 组织，不按来源业务线先分仓。
- 默认状态流转固定为：
  - `待响应`
  - `处理中`
  - `超 SLA`
  - `待回访`
  - `已关闭`
- 统一治理中心优先承接轻量处置动作；重型处理动作应跳回具体业务工作台完成。
- 投诉详情默认至少包含：
  - `案件摘要`
  - `关联对象`
  - `用户声音与证据`
  - `平台判断`
  - `处置动作`
  - `完整时间线`

### 医生 / 问诊运营

- 医生 / 问诊运营页至少从以下 4 个维度组织：
  - `需求量`
  - `转化`
  - `服务质量`
  - `供给状态`
- 热门医生或重点医生榜单应放在这个模块内，不放到首页总览。
- 医生排序优先按 `综合运营分`，不是按单一问诊量、收入或曝光量。
- 复诊贡献、复购贡献等指标属于机会识别，不应一律归类为风险。

## 十一、系统配置中心的产品语义

- 系统配置不是只放几个开关，它是后台的底层规则中心。
- 系统配置中心至少应承接：
  - `系统字典`
  - `模板配置`
  - `默认阈值`
  - `模块开关`
  - `默认文案位`
  - `角色与权限配置`
  - `风险阈值默认值`
  - `导出 / 留存基础配置`
  - `审核策略基础参数`
- 系统配置承担的是底层规则维护，不应退化成普通业务运营页。

## 十二、视图、筛选、导出与跳转语义

高频工作台默认必须支持视图化使用，而不是每次重填筛选条件。

执行约束：

- 高频工作台优先支持 `个人保存视图`。
- 保存视图至少保存：
  - 筛选条件
  - 排序方式
  - 可见列
- 用户重新进入页面时，应优先恢复：
  - 上次使用视图
  - 或默认个人视图
- 系统必须内置预设视图：
  - 治理中心按 `SLA / 阶段`
  - 履约中心按 `风险 / 时效`
  - 商品管理按 `经营表现 / 状态`
- 导出 V1 先覆盖工作台列表，不要求先做图表导出。
- 导出语义固定按“当前筛选结果导出”，不能导出与当前页面状态不一致的全集。
- 导出主要服务内部协同、复盘、复核与交接，不应先围绕外部报表系统设计。
- 需要一个轻量 `导出中心`：
  - 全局顶部有统一入口
  - 列表页局部区域可展示最近导出记录

## 十三、对象级跳转与处理闭环

- 宏观看板必须能下钻到专题工作台。
- 工作台条目必须能进入对象详情。
- 详情页必须保留回看日志、版本或时间线的能力。
- 轻量处理优先在当前工作台完成，复杂处理再跳转到专题页。
- 看板、列表、详情、日志之间必须形成闭环，不能停留在“只能看趋势，不能处理对象”的半成品后台。
