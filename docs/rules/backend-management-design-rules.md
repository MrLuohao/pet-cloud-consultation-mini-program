# Backend Management Design Rules

适用范围：`design/backendManagement.pen` 与后台管理端 `.pen` 画布协作。

## 一、分层边界

- `design/backendManagement.pen` 是管理后台页面设计的唯一正式事实源。
- 本文件只定义后台 `.pen` 画布层规则：
  - 信息架构
  - 页面骨架
  - 工作台形态
  - 列表 / 抽屉 / 详情的交互结构
  - 视觉系统
  - 组件系统
  - 动效基调
- 页面内容语义、模块职责、对象详情结构、榜单含义、视图 / 导出产品语义，统一以下沉到 `docs/rules/backend-management-product-rules.md` 为准。
- 指标公式、阈值、SLA、权限、日志、导出留存等后端实现契约，统一以下沉到 `docs/rules/backend-development-rules.md` 为准。

## 二、后台定位与角色边界

- 管理后台统一按桌面端 Web 后台理解，不按小程序页面、移动端后台或平板后台理解。
- 当前后台第一阶段按 `平台主后台` 理解，不按独立商家后台或双角色后台理解。
- 当前后台定位统一为：
  - `运营中台`
  - `内容配置中台`
  - `审核决策后台`
- 当前后台不是传统 ERP，也不是只看图表的大屏驾驶舱。
- 如果后续新增独立商家后台、商家工作台或双角色切换，必须先重定义信息架构，再补画独立链路。

桌面 Web 约束：

- 页面比例、分栏、表格宽度、侧栏结构、筛选区和操作区都必须按 Web 后台习惯组织。
- 不得沿用小程序单列长页、底部 Tab、移动端安全区或窄屏状态栏思路。
- 如果后续要补画移动运营页或窄屏适配，必须作为独立设计对象处理，不能直接复用当前正式后台页。

## 三、后台 V1 信息架构

当前后台 V1 固定按以下 7 个一级模块组织：

1. 运营总览
2. 商品管理
3. 内容配置
4. 社区审核
5. 订单履约
6. 用户 / 服务运营
7. 系统配置

执行约束：

- `用户 / 商家` 统一更名为 `用户 / 服务运营`。
- 左侧导航必须按业务域分组，不按零散页面堆叠。
- 首页默认是 `运营总览`，不是纯导航落地页。
- `用户 / 服务运营` 下当前至少包含以下二级页面：
  - `用户 360 运营台`
  - `统一治理中心`
  - `医生 / 问诊运营`
- 如果后续扩展新能力，优先并入现有业务域，不要无限新增一级导航。

## 四、当前补画优先级与节奏

后台第一批正式画布默认优先补齐：

1. `运营总览 Dashboard`
2. `商品内容配置中心`
3. `社区审核中心`

后续优先顺序建议：

1. `首页精选配置 / 发布排期`
2. `订单履约中心`
3. `用户 360 运营台`
4. `统一治理中心`
5. `医生 / 问诊运营`
6. `系统配置中心`

补画节奏规则：

- 连续补画正式后台页面时，默认每完成约 3 个页面暂停一次，先确认方向再继续。
- 未确认前，不要一次性把整个后台所有模块全部铺完。
- 如果导航分组、首页形态或核心工作流发生明显变化，必须先更新本文件，再继续补画。

## 五、页面骨架规则

### 运营总览页

- 首页第一屏必须同时体现：
  - 核心指标
  - 待处理事项
  - 快捷入口
  - 风险 / 异常提醒
- 首页不能只剩一排 KPI 卡，也不能只做导航入口宫格。
- 首页第一屏结构固定为：
  - 第 1 行：4 张一级业务卡
  - 第 2 行：4 张较轻的健康指标卡
  - 中段：3 个工作区块
  - 下段：2 个分析区块
- 首页只保留 1 个主趋势图，不并列堆多个同级折线区。
- 首页只保留 1 个首页榜单位，不并列放多个排行榜。

### 配置型页面

- 配置型页面统一按以下骨架组织：
  - 候选池 / 配置目录
  - 配置编辑区
  - 预览区
  - 状态区
- 不允许只画成单列长表单。
- 预览区必须可见，不能把“配置结果长什么样”完全藏到发布后验证。
- 预览统一按“小程序前台预览 + 映射说明”理解。
- 如果配置结果会影响前台首页、商品详情、精选位或推荐位，画布中必须直观看见配置结果会落到哪里。

### 审核型页面

- 审核中心默认按以下骨架组织：
  - 队列列表
  - 内容预览
  - AI 判定 / 人工决策面板
- 禁止把审核中心画成普通内容列表页。
- 审核页应显式保留：
  - 命中规则解释区
  - 人工复核区
  - 申诉 / 回查入口
  - 处置历史或时间线入口

### 列表 / 工作台页面

- 任务型、风险型、待办型工作台优先使用 `卡片队列`，而不是默认大表格。
- 资产型、日志型、记录型页面优先使用 `表格列表`。
- 列表页默认包含：
  - 顶部筛选条
  - 主操作区
  - 主体列表 / 表格
  - 详情抽屉、侧栏或局部操作面板
- 高频筛选放在第一行，次级筛选默认折叠进高级筛选区。
- 能在当前工作流内闭环的动作，优先使用抽屉、侧栏或局部面板，不要频繁整页跳转。
- 轻量详情默认优先抽屉；复杂对象的 360 视图再使用完整详情页。

## 六、视觉系统

后台视觉方向固定为 `深色外壳 + 浅色内容区`。

执行约束：

- 深色壳层优先放在：
  - 左侧导航
  - 顶部壳层
  - 少量重点摘要卡
- 内容区背景应使用偏冷、干净的浅灰白，不要使用大面积脏灰或泛紫灰。
- 常规卡片默认使用白色或近白背景，层级依赖留白、描边和轻阴影区分，不靠多层灰块堆叠。
- 需要减少大量近似灰白底、近似描边、近似投影同时出现造成的“发灰、发脏、蚊子感”。
- 品牌强调色继续沿用偏深的靛蓝 / 紫色体系，但使用必须更克制，只承担激活、主 CTA、图表主线和少量焦点状态。
- 语义色统一使用低饱和版本：
  - 成功色
  - 警告色
  - 错误色
- 不允许把品牌色大面积铺满整个内容区。
- 不允许发展成彩虹后台、霓虹面板或强毛玻璃堆叠。

视觉禁止项：

- 禁止普通白底模板后台感。
- 禁止无节制渐变、发光、毛玻璃叠加。
- 禁止所有卡片都做成强强调色卡片。
- 禁止为了“科技感”牺牲可读性与操作效率。

## 七、组件系统

### 圆角与间距

- 全局圆角只使用：
  - `32`
  - `24`
  - `18`
  - `pill`
- 全局间距节奏只使用：
  - `8`
  - `12`
  - `16`
  - `24`
  - `32`

### 文字系统

- 页面标题：`28`
- 大卡标题：`20`
- 分区标题：`16`
- 正文 / 按钮 / 表格主文本：`14`
- 次级正文：`13`
- 元信息：`12`
- `11` 基本禁止用于正文，只允许极少数图表刻度或附属注记使用。

### 列表与面板

- 主表格行高统一按 `52`。
- 紧凑型表格行高统一按 `44`。
- 快速操作抽屉宽度默认约 `440`。
- 深度处理抽屉宽度默认约 `680`。

### 按钮体系

- 全局按钮只保留 3 个层级：
  - 主按钮
  - 次按钮
  - 文字 / 轻按钮
- 不额外扩展第四、第五层视觉等级。

## 八、动效基调

- 动效统一为稳定、顺滑、低存在感。
- 动效只服务于：
  - 层级切换
  - 抽屉 / 面板展开
  - 状态反馈
  - 数字刷新
- 不做持续闪烁、漂浮粒子、过度旋转或强存在感装饰动画。

## 九、后台设计与实现协作规则

- 后台页面进入代码前，必须先以 `design/backendManagement.pen` 的正式页面为准。
- 如果后台正式画布尚未确认，不能提前进入后台前端实现阶段。
- 后台前端后续如果建立 schema，也必须从当前正式 `.pen` 结构化读取，不得凭截图或口头描述直接落代码。
- 如果当前 `.pen`、旧摘要、旧原型或旧代码之间冲突，以最新正式后台画布为准。
- 如果发现当前后台模块划分、字段表达或工作流与既有规则冲突，应先调整规则与画布，不允许让旧接口反向定义后台页面。

## 十、偏离规则的前提

只有在以下情况，才允许偏离本文件中的默认后台设计方向：

- 平台能力明确受限
- 性能或合规要求限制
- 后台角色模型发生正式变化
- 用户明确确认更改后台信息架构或视觉方向

若出现上述情况，必须先更新本文件，再继续补画后台页面。
