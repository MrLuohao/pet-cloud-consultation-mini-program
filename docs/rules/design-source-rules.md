# Design Source Rules

适用范围：`design/` 下全部 `.pen` 设计源与基于画布的协作。

## 一、设计源铁律

- `design/frontendMobileScreens.pen` 是消费端页面的正式事实源。
- `design/backendManagement.pen` 是管理后台页面的正式事实源。
- 所有与设计相关的判断都必须以相关 `.pen` 的最新实现为准。
- 如果 `.pen` 已发生修改并形成新版本，后续同步、补画、补 schema、补代码时必须站在新版本继续推进。
- 绝对不允许拿旧内容、旧截图、旧 schema、旧摘要或旧代码去覆盖或回退新的 `.pen`。
- 消费端页面设计细则统一以下沉到 `docs/rules/frontend-mobile-design-rules.md` 为准。
- 管理后台页面形态与视觉细则统一以下沉到 `docs/rules/backend-management-design-rules.md` 为准。
- 管理后台模块职责与页面语义统一以下沉到 `docs/rules/backend-management-product-rules.md` 为准。
- 前端与后端实现细则分别以下沉到 `docs/rules/frontend-implementation-rules.md` 与 `docs/rules/backend-development-rules.md` 为准。

## 二、画布到实现链路

- 页面设计变化必须先更新 `.pen` 画布，再进入 schema 和代码阶段。
- schema 同步必须通过 Pencil MCP / pencli 直接读取节点结构完成。
- 截图只用于最终校验，不得把视觉识别当成 schema 生成依据。
- 如果 MCP 当前读到的 `.pen` 与旧摘要、旧 schema、旧截图不一致，必须立即以当前 `.pen` 为准校正相关文档，而不是反向把 `.pen` 改回旧状态。
- 如果目标 `.pen` 仍在调整，或用户尚未明确确认“可以开始落代码”，与该页面强相关的 schema 与页面实现不得提前启动。

## 三、画布修改边界

- 设计不是参考图，必须按最终画布理解。
- 页面结构、模块顺序、弹层形态、状态切换、地图交互、成功反馈都先看 `.pen`。
- 未确认前，不得自行新增、删除、重排模块。
- 新增或补画正式页面时，必须按当前功能模块分区摆放，并尽量贴近实际调用链、入口页或相邻功能页，不能把同一模块页面散落到无关区域。
- 如果必须偏离 `.pen`，只能因为平台能力、性能、合规或安全限制，并且必须先确认。

## 四、画布评审标准

- 进入 `.pen` 设计阶段时，默认必须以“产品 + 设计 + 实现”的复合视角进行判断，不得只做机械画图。
- 每次画图时都必须主动参考成熟产品中已经验证过的优秀交互、信息组织和视觉处理方式，但不能脱离当前项目的真实调用链、功能语义和既有视觉体系。
- 需要主动审视：
  - 信息层级
  - 操作语义
  - 情绪氛围
  - 视觉秩序
  - 用户决策路径
- 当用户要求继续调整 `.pen` 画布时，默认先做专业设计复盘，再进入具体改画布动作。
- 如果发现结构、主次关系、按钮语义或视觉焦点存在问题，必须先明确指出，不得直接照旧微调。

## 五、协作节奏与确认规则

- 连续补画正式页面时，默认每完成约 3 个页面就暂停一次，先让用户确认是否需要调整，再继续下一批。
- 未经过这一轮确认，不要连续批量补完整个模块，避免后续集中返工。
- 如果用户明确要求连续推进更多页面，可以按用户最新口径执行。
