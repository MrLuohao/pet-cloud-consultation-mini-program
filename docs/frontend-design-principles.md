# Frontend Design Principles

适用范围：`PetCloudConsultation_Frontend` 小程序页面设计、重构、视觉优化，以及后续与 AI 协作时的统一约束。

快速入口：如需先看项目当前统一规则与优先级，优先阅读 `docs/project-rules.md`。

## 设计目标

这个项目的前端页面应当追求以下气质：

- 简约
- 柔和
- 有呼吸感
- 克制但不冷淡
- 高级感来自秩序和细节，而不是装饰堆砌

不追求：

- 花哨的运营风
- 彩虹式功能面板
- 大量渐变、贴纸、装饰图形
- 为了“显得高级”而做得过度空洞

## 总体视觉方向

项目整体更适合这条主线：

`柔和、克制、轻社交、轻服务的宠物产品风格`

这意味着：

- 页面应该友好、有温度，但不能幼稚
- 页面可以带一点宠物产品的亲和力，但不要走强可爱路线
- 功能信息要清楚，但不能像后台或运营页

## 场景化色调统一规则

允许不同业务场景做轻度色调偏移，但必须保持同一套设计体系，不允许每个页面各自发展为不同产品。

### 1. 品牌内容页

适用页面：

- 首页
- 社区
- 我的
- AI 诊断

主基调：

- 紫白系
- 轻粉感
- 更柔和的品牌氛围

作用：

- 承担品牌感
- 承担内容感与亲和力
- 强化宠物产品的轻服务气质

### 2. 交易确认页

适用页面：

- 商城
- 商品详情
- 购物车
- 结算
- 支付相关页面
- 地址管理
- 地址编辑

主基调：

- 暖灰白
- 米白
- 低饱和中性色

作用：

- 提升稳定感、可信感与高级感
- 弱化传统电商促销感
- 让交易链路更安静、更克制

### 3. 成功反馈色

绿色只用于成功状态与完成反馈：

- 支付成功
- 操作成功
- 状态确认

禁止把绿色扩展为页面主色。

### 4. 全局统一不变项

无论页面属于哪种场景，都必须统一：

- 留白节奏
- 圆角体系
- 文本层级
- 轻描边逻辑
- 浅背景策略
- 低饱和控制

结论：

`颜色可以按场景轻分化，但设计体系必须统一。`

## 核心设计原则

### 1. 卡片要轻，不要厚

- 可以使用白底、圆角、轻描边
- 少用重阴影、强描边、厚重边界
- 卡片应该是“内容容器”，不是“功能模块盒子”
- 如果内容已经足够清楚，优先靠留白分层，而不是靠更重的卡片边界分层

### 2. 强调色要少而准

- 主强调色以柔和紫色为主
- 蓝色只能做少量辅助，不应成为第二主色
- 避免一个入口一个颜色
- 避免高饱和、大面积渐变

推荐理解：

- 紫色负责品牌感和轻重点
- 中性色负责绝大部分信息表达
- 页面里的“彩色面积”必须被严格控制

### 3. 顶部结构做减法

- 每个页面顶部只允许有一个真正的强焦点
- 搜索、Tab、筛选、话题、Banner 这类元素不能同时都很强
- 用户应当尽快进入页面真正的内容区

判断标准：

- 用户进入页面时，第一眼应该知道“这里是什么”
- 第二眼应该直接看到“这里最重要的内容”
- 不要先让用户阅读一大段导航结构

### 4. 操作区要轻反馈化

- 点赞、评论、收藏、分享、设置、发布等操作，不要做成控制面板
- 操作应该“随时可用”，但不应长期抢主视觉
- 点赞/评论这类高频动作可以略强
- 收藏/分享/设置等低频动作应弱化

目标效果：

- 操作更像内容反馈
- 不像一排平均用力的功能按钮

### 5. 高级感来自文字层级

优先通过以下方式提升质感：

- 标题字重
- 主副信息颜色层级
- 文本行距
- 模块间距
- 信息密度控制

不要优先依赖：

- 更大的图标
- 更多图标
- 更重的渐变
- 更复杂的背景装饰

### 6. 允许微炫技，但只做细节炫技

项目允许有“高级感”的动态与交互细节，但必须遵守以下原则：

- 可以炫手感，不能炫场面
- 可以有微交互，不能让页面到处都在动
- 动效必须服务于理解、反馈和质感，不能只是为了热闹

适合保留的动态表达：

- 按钮点击时的轻微缩放、回弹
- Tab 切换时的丝滑过渡
- 搜索框聚焦时边框、背景、阴影的轻变化
- 卡片按下时的轻微下沉
- 模块进入时的轻透明度变化和短位移
- 主 CTA 上极轻的高光掠过
- 关键卡片在激活状态下的轻微强调

不适合保留的动态表达：

- 大面积粒子、浮球、光斑同时出现
- 多个模块各自动效语言不一致
- 高频、长时间、强存在感的背景动画
- 过度依赖渐变流动、悬浮漂移、网格扫描来制造“高级感”

## 设计还原执行规则

后续所有开发工作，必须把 `.pen` 设计画布视为最终设计基准。

### 1. 1:1 还原原则

开发实现必须与 `.pen` 画布保持 1:1 对齐，包括但不限于：

- 页面结构
- 模块顺序
- 布局比例
- 间距
- 圆角
- 字体层级
- 颜色
- 图标
- 动效触发关系
- 状态切换方式
- 交互反馈与特效

### 2. 禁止自行发挥

未经明确确认，禁止开发过程中自行：

- 新增模块
- 删除模块
- 修改模块顺序
- 替换视觉风格
- 擅自简化交互
- 擅自弱化或删除特效
- 擅自新增“自认为更合理”的功能入口

### 3. 禁止偷工减料

如果 `.pen` 中已经定义了：

- 弹层形态
- 转场方式
- 支付验证流程
- 成功反馈样式
- 地图选点方式
- 自动识别回填逻辑

则实现阶段必须按设计内容落地，不允许因为实现麻烦就退化成普通页面、普通弹窗、普通表单或普通提示。

### 4. 开发判断优先级

开发时的优先级必须是：

1. `.pen` 画布
2. 已确认的设计文档
3. 现有代码结构

这意味着：

- 代码要服务设计
- 不能反过来让设计迁就已有低质量实现

### 4.1 系统状态栏与导航栏的实现约束

针对微信小程序：

- `.pen` 里出现的时间、信号、电量等系统状态栏内容，只用于还原画布上下文，不直接输出到页面代码
- 默认使用微信原生导航栏承接标题、返回与系统顶部空间
- 未经确认，不允许为了“看起来更像设计稿”而新增伪状态栏或自绘导航栏
- 若确需自定义导航结构，必须先确认原因，并同步更新规则文档与 schema 约束

### 5. 如需偏离，必须先确认

只有以下情况允许偏离 `.pen`：

- 平台能力明确不支持
- 性能成本不可接受
- 合规或安全限制
- 已与产品 / 设计明确确认调整

否则一律按 `.pen` 实现。

结论：

`后续开发必须严格按照 .pen 画布 1:1 还原，不允许画蛇添足，也不允许偷工减料。`

一句话规则：

`动效应该让用户觉得顺、细、稳，而不是让用户觉得页面很会动。`

## 页面级别原则

### 我的页

适合方向：

- 安静卡片流
- 个人信息为第一焦点
- 订单/服务/宠物有清晰优先级
- 设置等低频入口弱化处理

避免：

- Hero 区过度渐变
- 彩色功能面板
- 所有入口都同等抢眼

### 社区页

适合方向：

- 轻社交型信息流
- 顶部轻，内容优先
- 帖子卡片轻容器化
- 互动栏像反馈区，不像工具栏
- 普通宠物帖头像要嵌入身份信息块内部，右侧三行固定为宠物名、`品种 · 年龄`、个性签名
- 专家/医生帖保持更克制的两行头部，不额外插入年龄胶囊或宠物人格信息

避免：

- 搜索、Tab、热门话题同时很重
- 帖子像标准模板卡片
- 四个互动按钮同等用力
- 把年龄拆成第一行标签
- 头像悬空在信息块之外，造成头部脱节

### 服务/商城类页面

适合方向：

- 信息清楚
- 卡片轻
- 视觉重点集中在少数主信息
- 商品、服务、价格层级清楚

避免：

- 电商促销风过重
- 满屏颜色竞争
- 多个运营焦点同时出现

## 组件级别原则

### 搜索框

- 可以非常轻
- 不一定必须保留左侧图标
- 更像入口，不像表单控件

### Tab

- 作为主导航时可以保留强调
- 不要同时叠加强下划线、强色块、强背景
- 更适合用字重和颜色建立主次

### 标签

- 标签应是轻信息，不是按钮
- 宠物标签、话题标签、身份标签都应克制
- 除非确实需要强调，否则不要做成大色块徽章

### 悬浮按钮

- 保留可用性
- 减少视觉压迫
- 更像工具，不像主视觉
- 可加入轻微按压和出现动效，但不要做成持续抢眼的动态焦点

### 互动栏

- 主互动：点赞、评论
- 次互动：收藏、分享
- 视觉上不应四等分平均用力

## 微交互设计原则

### 总原则

- 动效优先服务于状态变化
- 动效时间要短
- 动效曲线要柔和
- 动效不应破坏信息阅读

### 推荐用法

#### 1. 点击反馈

- 按钮：轻缩放、轻回弹
- 卡片：轻压下、轻阴影变化
- 图标按钮：轻透明度或轻位移变化

#### 2. 状态过渡

- Tab 激活态平滑切换
- 搜索框 focus 状态平滑过渡
- 点赞/收藏状态切换时轻提示

#### 3. 模块进入

- 首屏模块可以有很轻的顺序显现
- 推荐使用透明度 + 轻位移
- 避免大范围飞入、翻转、旋转

### 时长建议

- 触达反馈：150ms - 220ms
- 状态切换：180ms - 260ms
- 模块进入：240ms - 360ms

### 风格建议

- 使用柔和 ease 曲线
- 避免过强 bounce
- 避免长时间循环动画
- 如果一个动画会持续超过 1 秒，必须谨慎评估是否真的必要

## 与 AI 协作时的推荐提示词约束

后续让 AI 帮忙改页面时，建议明确加上这些约束：

- 不改变现有功能，只调整排版和视觉层级
- 页面要简约，但不能空
- 高级感来自秩序、留白和文字层级，不来自装饰堆砌
- 减少彩色块和强渐变
- 一个页面只保留一个主焦点
- 卡片边界要轻
- 操作区像反馈，不像工具栏
- 宠物产品要有温度，但不要幼稚
- 允许克制型微交互，但不要使用大场面炫技动画

## 可直接复制的 AI 提示词模板

### 模板 1：页面视觉优化

适用于：不改功能，只优化某个页面的排版和视觉层级。

```text
请基于项目文档 `docs/frontend-design-principles.md` 中的设计原则，帮我优化这个页面。

要求：
1. 不改变现有功能和信息结构，只调整排版、视觉层级和样式表达
2. 页面要简约，但不能空
3. 高级感来自秩序、留白、字重层级和克制的强调色，不来自复杂装饰
4. 减少彩色块、强渐变、厚重卡片和多余图标
5. 页面只保留一个主焦点，其余内容做清晰的主次分层
6. 操作区像反馈，不像工具栏
7. 宠物产品要有温度，但不要幼稚，不要过度可爱化

请先分析这个页面当前存在的问题，再给出优化方向，然后再输出修改方案。
```

### 模板 2：页面设计稿讨论

适用于：先讨论方向，不急着改代码。

```text
我想讨论这个页面该怎么优化，请先遵守 `docs/frontend-design-principles.md`。

目标：
- 保留现有功能
- 做得更简约、更有高级感
- 不要做成花哨运营风

请按下面顺序回答：
1. 当前页面最影响高级感的 3 个问题
2. 最值得优先改的部分
3. 如果只做一轮优化，你建议怎么改
4. 哪些地方不建议继续改，避免过度设计
```

### 模板 3：落地实现前的设计约束

适用于：准备让 AI 直接写前端代码或改样式。

```text
请先阅读并遵守 `docs/frontend-design-principles.md`，再进行实现。

实现要求：
- 不改变现有功能逻辑
- 优先做减法，不要为了“更丰富”增加无关装饰
- 卡片边界要轻，强调色要少
- 顶部结构做减法，让用户尽快进入主要内容
- 互动区域做轻反馈化处理
- 页面保持可实现、可维护，不要做只适合设计稿展示的样式

输出时请说明：
1. 你保留了哪些原有功能
2. 你重点优化了哪些视觉层级
3. 你如何保证风格符合项目设计原则
```

### 模板 4：让 AI 自检产出质量

适用于：AI 改完页面后，要求它自己复盘。

```text
请根据 `docs/frontend-design-principles.md` 对你刚才给出的页面方案做一次自检。

请逐项回答：
1. 这个页面的主焦点是否明确？
2. 顶部是否过重？
3. 是否有过多颜色竞争？
4. 卡片是否过厚？
5. 操作区是否太像工具栏？
6. 页面是否足够有呼吸感？
7. 是否存在“为了高级而做空”或“为了丰富而做乱”的问题？

如果有问题，请直接给出修正建议。
```

## 快速检查清单

每次改页面前或改完后，都可以用下面这份清单自检：

- 这个页面的第一视觉焦点是否只有一个？
- 顶部区域是否过重？
- 是否用了过多颜色？
- 卡片是否过厚、过重？
- 操作按钮是否太像工具栏？
- 文本层级是否清楚？
- 页面是否有足够呼吸感？
- 是否为了“高级”而把页面做空了？
- 是否为了“丰富”而把页面做乱了？

如果以上任一问题回答为“是”，优先先做减法。
