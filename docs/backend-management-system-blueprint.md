# Backend Management System Blueprint

更新时间：2026-03-29

适用范围：后台管理系统后续补画、重构、页面拆分、看板规划、对象处理闭环设计。

说明：

- 本文档用于完整沉淀当前已经统一过的后台管理系统设计结论，方便新会话直接续接。
- 本文档不是长期规则仲裁文件；如果后续与正式规则文件冲突，以 `docs/rules/` 下对应规则为准。
- 本文档当前主要承担“全量设计梳理稿”的作用，帮助后续直接进入 `design/backendManagement.pen` 的补画与重构。

## 一、后台定位

- 这是一个桌面端 Web 平台主后台，不是小程序后台，不是商家后台，也不是传统 ERP。
- 它的核心定位是：
  - 运营中台
  - 内容配置中台
  - 审核决策后台
- 它不是纯 BI 看板，也不是只做列表处理；必须同时承接“宏观经营判断 + 微观对象处理”。
- 后台 V1 的经营主线不是纯商品库管理，而是 `内容带成交`。
- 归因口径默认按 `点击后归因` 理解，不按曝光归因理解。
- 后台的价值不是“看数据”，而是“配置、发布、审核、监控、处理对象”。

## 二、后台一级信息架构

当前后台固定为 7 个一级模块：

| 一级模块 | 作用 |
| --- | --- |
| 运营总览 | 全局经营判断、今日优先级、跨域风险入口 |
| 商品管理 | 商品列表、上下架、商品详情内容配置、商品经营下钻 |
| 内容配置 | 首页精选、内容位、排期、预览、发布、回滚 |
| 社区审核 | 审核队列、申诉回查、误杀纠偏、社区健康洞察 |
| 订单履约 | 发货、物流、售后、履约风险、升级治理 |
| 用户 / 服务运营 | 用户 360、统一治理、医生 / 问诊运营 |
| 系统配置 | 系统模板、字典、角色、阈值、模块开关、默认文案 |

必须明确：

- `用户 / 商家` 已经改成 `用户 / 服务运营`
- `统一治理中心` 和 `医生 / 问诊运营` 不是一级导航，是 `用户 / 服务运营` 下的二级页

## 三、后台最终应该补成的核心页面

V1 至少应该形成下面这批正式页面：

| 页面 | 类型 |
| --- | --- |
| 运营总览 Dashboard | 总览页 |
| 商品列表 / 上下架管理 | 列表工作台 |
| 商品详情内容配置工作台 | 配置工作台 |
| 首页精选配置 / 发布排期 | 配置工作台 |
| 社区审核中心 | 审核工作台 |
| 订单履约中心 | 风险 / 时效工作台 |
| 用户 360 运营台 | 对象运营页 |
| 统一治理中心 | 跨域治理工作台 |
| 医生 / 问诊运营 | 专题运营页 |
| 系统配置中心 | 系统配置页 |

说明：

- 当前即使已经存在部分页面，也不代表页面内容不能重构。
- 只要不符合已经统一后的结构，就应按本蓝图重构，而不是迁就旧页面内容。

## 四、全局设计原则

- 每个模块都要同时有宏观看板和微观对象处理入口。
- 宏观看板负责：
  - 看趋势
  - 看异常
  - 看结构
  - 看优先级
- 微观处理负责：
  - 进入具体对象
  - 看详情
  - 做动作
  - 看日志
  - 看时间线
  - 回到列表
- 不能只有图表没有处理动作。
- 不能只有表格没有经营判断。
- 列表 / 工作台页优先局部闭环，不要所有动作都整页跳来跳去。
- 高频任务页优先 `卡片队列`。
- 资产 / 日志 / 明细页优先 `表格`。
- 轻量详情优先 `抽屉`。
- 复杂 360 详情再用整页。

## 五、运营总览应该怎么设计

运营总览不是简单 KPI 宫格，而是“今天先处理什么”的总工作台。

首页第一屏固定结构：

- 第 1 行：4 个一级经营卡
- 第 2 行：4 个健康指标卡
- 中段：3 个工作区块
- 下段：2 个分析区块

首页固定的 4 个一级经营卡：

- 支付成交总额
- 内容带成交额
- 精选点击率
- 归因支付订单数

首页固定的 4 个健康指标：

- 发货准时率
- 投诉率
- 超 SLA 单量
- 售后平均处理时长

首页中段固定的 3 个区块：

- 今日优先处理
- 风险 / 异常诊断
- 热门商品榜

首页下段固定的 2 个区块：

- 内容转化链主趋势图
- 最近关键动作

首页明确限制：

- 首页只保留 `1 个主趋势图`
- 首页只保留 `1 个首页榜单`
- 首页榜单固定优先 `热门商品榜`
- `热门商品榜` 的排序口径固定按 `内容带成交`
- `热门医生榜` 不应该放首页，应该放到 `医生 / 问诊运营`
- 用户复购率、复诊贡献这类深专题指标，不该挤在首页首屏

首页快捷动作固定优先：

- 新建精选排期
- 处理高优投诉
- 审核高风险内容
- 查看履约异常

首页每个异常项都必须能跳转：

- 异常卡片要跳到对应工作台
- 榜单条目要跳到对应对象或专题页
- 最近关键动作要能回到原对象或原版本记录

## 六、商品管理应该怎么设计

商品管理不是只有一个商品列表。

它至少拆成两个核心页面：

1. 商品列表 / 上下架管理
2. 商品详情内容配置工作台

商品列表要承接：

- 搜索、筛选、状态管理
- 上下架动作
- 经营表现摘要
- 热门 / 下滑 / 高潜商品识别
- 跳到商品详情内容工作台

商品详情内容配置工作台必须是结构化模块，不是富文本大文本框。至少要有：

- 首图 / 图集
- 核心卖点
- 功效说明
- 使用方式
- 规格参数
- 成分 / 材质
- 服务承诺
- FAQ
- 相关推荐
- 评价摘要

这里的本质是：

- 商品详情页内容以后要通过后台动态配置
- 商品详情页的模块化展示要在后台可管理
- 上下架、推荐、详情内容维护，要形成统一闭环，不是分散在很多临时页里

## 七、内容配置应该怎么设计

内容配置是这个项目里非常关键的一类后台，不是简单文案管理页。

重点对象包括：

- 首页精选
- 详情推荐位
- 活动内容位
- 系统内容位
- 动态跳转位

内容配置页必须具备这些能力：

- 草稿
- 预览
- 排期
- 发布
- 下线
- 版本
- 回滚

页面结构必须清晰承接：

- 候选池
- 当前在线内容
- 即将生效内容
- 历史版本
- 前台预览
- 映射说明

必须明确：

- 首页精选、商品详情精选、推荐位等，以后都要能通过后台直接上架、下架、替换、推送
- 这些能力不能写死在前端里

## 八、社区审核应该怎么设计

社区审核不是普通内容列表，而是“审核工作台 + 内容健康洞察”。

它至少要同时承接：

- 待审核
- 待复核
- 申诉回查
- 命中规则解释
- AI 判定结果
- 人工复核
- 处置历史
- 手动纠偏
- 轻量用户治理动作

社区健康洞察至少要有这些维度：

- 发帖量
- 举报率
- 拦截率
- 申诉率
- 误杀纠偏率

社区审核页的核心要求是：

- 既能看风险趋势
- 也能处理某条具体内容
- 还能看 AI 为什么这么判

不能把它画成一个只会“通过 / 拒绝”的审核流水页。

## 九、订单履约应该怎么设计

订单履约不是订单状态表，而是 `风险 + 时效` 工作台。

它要围绕这些经营问题：

- 发货准时
- 物流异常
- 售后超时
- 是否升级到治理中心

它要能回答：

- 今天哪些单最该先处理
- 哪类异常正在变坏
- 哪些售后单正在超时
- 哪些问题已经从履约层升级成投诉 / 治理问题

所以这个页面的排序与组织应该按：

- 风险等级
- SLA
- 用户影响
- 时效优先级

而不是简单按创建时间倒序。

## 十、用户 / 服务运营应该怎么设计

这个模块不是 CRM 增长页，它是服务 / 风险优先的对象运营模块。

它下面至少有 3 个核心页。

### 1. 用户 360 运营台

- 不是看用户增长漏斗
- 是看一个用户值不值得重点服务、有没有风险、最近发生了什么

至少包含这些区块：

- 基础资料
- 宠物摘要
- 订单 / 售后历史
- 投诉 / 反馈 / 举报历史
- 问诊 / 医生服务历史
- 社区行为 / 风险标签
- 最近关键动作

### 2. 统一治理中心

- 这是跨域投诉 / 风险 / 服务事件的总治理页
- 它不是按来源业务线分仓，而是按 `紧急度 / SLA` 组织

固定状态流转：

- 待响应
- 处理中
- 超 SLA
- 待回访
- 已关闭

它更像总分诊台，轻量动作在这里做，重处理再跳回业务工作台。

投诉详情固定结构：

- 案件摘要
- 关联对象
- 用户声音与证据
- 平台判断
- 处置动作
- 完整时间线

### 3. 医生 / 问诊运营

- 这是问诊业务的专题运营页
- 它不该挂在首页榜单里

至少按 4 个维度组织：

- 需求量
- 转化
- 服务质量
- 供给状态

这个页面里需要重点看：

- 哪些医生综合运营分高
- 哪些医生响应慢
- 哪些医生投诉率高
- 哪些医生复诊贡献高
- 哪个时段供给紧张

必须明确：

- 医生排序不能只按问诊量
- 也不能只按收入
- 必须按 `综合运营分`

## 十一、系统配置应该怎么设计

系统配置不是只放几个开关。

它至少应该承接：

- 系统字典
- 模板配置
- 默认阈值
- 模块开关
- 默认文案位
- 角色与权限配置
- 风险阈值默认值
- 导出 / 留存基础配置
- 审核策略基础参数

它的定位是底层规则中心，而不是业务运营页。

## 十二、哪些看板必须有

全局必须有两类看板：

- 宏观看板
- 微观对象处理板

宏观看板的作用：

- 看趋势
- 看分布
- 看异常
- 看排名
- 看优先级

微观对象处理板的作用：

- 进具体对象
- 看详情
- 做动作
- 看日志
- 看时间线
- 回到列表

每个模块建议承接的核心看板维度如下。

### 运营总览

- 内容带成交趋势
- 精选点击趋势
- 支付成交趋势
- 发货 / 投诉 / 售后健康度
- 热门商品榜
- 今日优先处理板

### 商品管理

- 热门商品
- 下滑商品
- 高潜商品
- 商品状态分布
- 内容配置完成度

### 内容配置

- 精选位表现
- 内容位占用 / 空档
- 排期覆盖情况
- 当前生效版本
- 版本变更与回滚记录

### 社区审核

- 发帖量趋势
- 举报率趋势
- 审核拦截率
- 申诉率
- 误杀纠偏率
- 高风险内容队列

### 订单履约

- 发货准时率趋势
- 物流异常趋势
- 售后时长趋势
- 超 SLA 队列
- 风险订单队列

### 用户 / 服务运营

- 用户服务价值
- 投诉 / 风险概览
- 问诊活跃度
- 历史服务轨迹
- 关键动作时间线

### 医生 / 问诊运营

- 需求量趋势
- 转化趋势
- 服务质量趋势
- 供给状态趋势
- 医生综合运营分榜
- 风险 / 机会医生榜

## 十三、图表应该怎么用

我们并不是不要图表，而是不要“为了有图表而堆图表”。

统一原则：

- 首页只允许 1 个主趋势图
- 首页只允许 1 个首页榜单
- 饼图只适合做结构占比快照，不适合做首页主图
- 折线 / 面积图适合趋势
- 条形图适合排行榜和分类对比
- 堆叠条适合状态构成
- 表格 / 队列适合微观处理
- 看板必须服务决策，不是装饰

图表使用建议：

- 首页主图用折线 / 面积趋势图
- 榜单用条形排行或列表排行
- 状态分布可用轻量环图或堆叠条
- 微观详情不要滥用图表，应该优先对象信息、证据、时间线、动作

## 十四、页面跳转与处理闭环

后台必须形成完整闭环：

- 总览页的异常卡片跳到对应模块工作台
- 模块工作台的条目能进入对象详情
- 对象详情能看日志与时间线
- 从日志或时间线可以回看版本与当前对象
- 轻量动作优先在列表 / 工作台当前页面完成
- 重动作再跳转到专题页
- 统一治理中心只做轻量治理和分诊
- 重型处理动作回到具体业务域完成

必须明确：

- 后台不能做成“只能看趋势，不能处理对象”
- 也不能做成“只能处理对象，没有宏观判断”

## 十五、保存视图与导出

高频工作台要支持视图化使用。

必须支持：

- 个人保存视图
- 系统预设视图
- 重新进入页面恢复上次视图

保存视图至少记录：

- 筛选条件
- 排序方式
- 可见列

系统预设视图至少有：

- 治理中心：按 SLA / 阶段
- 履约中心：按 风险 / 时效
- 商品管理：按 经营表现 / 状态

导出 V1 统一原则：

- 先导出列表，不先导出图表
- 按当前筛选结果导出
- 主要服务内部复盘、协作、复核
- 需要一个轻量导出中心
- 导出中心要有全局入口和局部最近导出记录

## 十六、视觉与界面风格

固定方向：

- 深色外壳 + 浅色内容区
- 品牌强调色继续使用深靛蓝 / 紫
- 但要更克制
- 成功 / 警告 / 错误都用低饱和语义色

必须处理掉的问题：

- 现在有些区域灰白太多，层次太近，容易发灰、发脏、像很多小蚊子一样乱
- 后续要减少大量近似灰块、近似描边、近似阴影叠在一起的感觉

组件系统固定约束：

- 圆角只用 `32 / 24 / 18 / pill`
- 间距只用 `8 / 12 / 16 / 24 / 32`
- 页面标题 `28`
- 大卡标题 `20`
- 分区标题 `16`
- 正文 / 表格 / 按钮 `14`
- 次级正文 `13`
- 元信息 `12`
- `11` 号字基本禁用，仅保留极少图表刻度
- 表格主行高 `52`
- 紧凑行高 `44`
- 抽屉宽度默认 `440 / 680` 两档
- 按钮只保留 `3` 个层级

## 十七、哪些属于画布，哪些属于后端

属于当前 `.pen` 和产品规则要表达清楚的：

- 页面结构
- 模块划分
- 看板种类
- 对象详情结构
- 状态流转
- 跳转关系
- 页面动作语义
- 配置对象与前台落点映射

属于后端开发规则再去实现的：

- 指标公式
- 阈值
- SLA
- 权限边界
- 审计日志字段
- 导出留存
- 脱敏规则

也就是说：

- 画布先定义“要有什么、怎么处理、怎么跳”
- 后端再实现“怎么算、怎么存、怎么控权限”

## 十八、最终一句话总结

这套后台不是“几个零散页”的拼接，而是一个围绕 `内容带成交 + 服务风险治理 + 对象级处理闭环` 搭起来的平台主后台。

后续补画顺序应按这套来：

- 先把现有页面全部重构到这套口径
- 再补齐缺失页面
- 再统一跳转、详情、日志、版本、视图和导出语义
