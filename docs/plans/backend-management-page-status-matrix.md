# Backend Management Page Status Matrix

更新时间：2026-03-29（已同步后台原型设计阶段口径）

适用范围：`design/backendManagement.pen` 中已进入正式后台设计链路的页面画板，以及已在当前后台规则中明确命名、但尚未落成正式画板的优先页面。

不含范围：

- 设计系统组件区
- reusable 组件示例
- 纯参考组件或辅助画板

说明：

- 本文档统一使用以下字段：
  - `页面设计状态`
    - `已画完成`：已有正式后台页面画板，且当前画板反映最新讨论结果
    - `补画中`：已有正式后台页面画板，但仍在当前轮补画或结构调整中
    - `已规划未画`：已在后台规则中明确命名与排序，但尚未落成正式页面画板
    - `待复核`：已有页面画板，但当前方向、结构或角色定位仍需重新确认
  - `规则对齐状态`
    - `已对齐设计 / 产品规则`
    - `已对齐设计规则，待补产品细化`
    - `待补规则映射`
  - `实现准备状态`
    - `不进入实现`
      - 当前仍处后台原型设计阶段，不进入后台前端或后端实现
    - `待后端契约`
      - 页面骨架已较明确，但指标口径、权限、导出、SLA 或接口对象仍需先补契约
    - `可进实现`
      - 当前页画布与规则已确认，可以进入后台页面实现
    - `Blocked`
      - 仍缺正式画板，或关键规则尚未收敛
- 优先级含义：
  - `P0`：后台当前批次最高优先补画页
  - `P1`：高价值主链路页，建议紧随其后推进
  - `P2`：辅助页或可后置页
  - `Blocked`：当前被画板缺失或规则未细化阻塞
- 所有判断统一以以下最新事实为准：
  - `design/backendManagement.pen`
  - `docs/rules/backend-management-design-rules.md`
  - `docs/rules/backend-management-product-rules.md`
- 当前后台仍处原型设计阶段：
  - 现阶段不能把已有 Java / SQL / 旧后台代码视为后台页面已完成依据
  - 后台页面当前不走消费端那套 `.pen -> schema -> app.json` 接入链路
  - 本表主要跟踪“正式画板是否完成、规则是否收敛、是否具备进入实现条件”
- 当前 `design/backendManagement.pen` 强一致性事实：
  - 当前顶层正式后台页面画板共 1 个：`JCVX7 / 运营总览 Dashboard`
  - 顶层 `f:vtHps / halo: design system components` 为设计系统组件区，不纳入后台页面进度
- 后续执行要求：
  - 每次后台页面状态变化后，都要同步更新本文档
  - 至少同步：
    - `页面设计状态`
    - `规则对齐状态`
    - `实现准备状态`
    - `备注`

## 三、当前建议推进顺序

说明：

- 以下排序统一按“后台原型设计阶段”理解，不代表正式后台研发排期
- 后台正式实现必须以后续确认后的 `design/backendManagement.pen` 与规则文件为准

### P0

1. 继续把第一批正式后台页补齐到 3 个以内，再暂停复核一次整体方向：`商品内容配置中心`、`社区审核中心`
2. 默认先补“真正的下一张正式后台页”，不要把设计系统组件区误判成页面完成度
3. 在继续补画前，先以本表核对当前真实状态，避免把“规则里已命名”误写成“画板已存在”

### P1

1. `首页精选配置 / 发布排期`
2. `订单履约中心`
3. `用户 360 运营台`
4. `统一治理中心`
5. `医生 / 问诊运营`

### P2

1. `系统配置中心`

## 四、维护规则

- 后台新页面一旦进入正式设计链路，就要同步更新本表
- 页面状态发生变化时，至少同步更新：
  - 页面设计状态
  - 规则对齐状态
  - 实现准备状态
  - 对应 frameId
  - 优先级
  - 备注说明
- 如果某个后台页被确认合并、降级为抽屉 / 局部面板，或被移出当前阶段，也必须在本表中显式改掉
- 只要后台仍处原型设计阶段，`实现准备状态` 默认不能直接写成 `可进实现`
