## 3. 当前项目统一设计基调

整体气质：

- 简约
- 柔和
- 克制
- 有呼吸感
- 高级感来自秩序和细节

明确不做：

- 花哨运营风
- 强促销电商风
- 彩虹式功能面板
- 过度空洞的“伪高级”

## 4. 场景化色调规则

### 品牌内容页

适用：

- 首页

- 我的
- AI 诊断

基调：

- 紫白系
- 轻粉感

- 圆角体系
- 文本层级
- 轻描边逻辑
- 浅背景策略
- 低饱和控制

## 5. 开发不可违反的硬规则

- `.pen` 不是参考图，而是最终设计源
- 开发必须 1:1 还原 `.pen`
- 未确认前，不允许自行新增模块、删除模块、修改顺序
- 未确认前，不允许擅自简化弹层、转场、验证流程、地图交互、成功反馈特效
- 若 `.pen` 已定义关键交互，不能因为实现麻烦退化成普通页面、普通弹窗、普通表单或普通提示
- 如果必须偏离 `.pen`，只能因为平台能力、性能、合规或安全限制，并且必须先确认
- `.pen` 中的 OS chrome 只作为参考，不得在页面中伪造渲染系统状态栏文本
- 微信小程序默认使用原生导航栏，不得再把时间、信号、电量、系统返回区直接画死在页面里
- 除非有明确确认，否则不得使用 `navigationStyle: custom` 重做导航结构
- 社区普通宠物帖头部必须是“头像 + 宠物名 / 品种·年龄 / 个性签名”的同一身份块，禁止把头像悬空放成单独一行
- 社区普通宠物帖不得把年龄拆成第一行右侧胶囊，第二行必须保留 `品种 · 年龄`
- 社区专家/医生帖按两行身份信息实现：姓名；`相对时间 · 专业答疑`
- 商品详情页的规格、亮点文案、故事段落、使用建议必须按后端结构化字段驱动，禁止继续把这类内容写死在前端模板中
- 商品详情页直购链路必须透传用户当前选择的规格，禁止下单时无条件回退 `default_spec`
- `imageUrls` 只作为商品图库，不得冒充详情正文结构字段；正文展示顺序只能由 `storySections[].imageUrl` 等显式字段决定

### 5.1 本轮新增的页面级硬规则（2026-03-16）

以下规则已经在实际画布调整中确认，后续继续补画、补 schema 和落前端时必须直接遵守，不再退回旧判断。

- 宠物列表页 `我的宠物列表 My Pets`（`cKCx8`）必须按“画廊式大卡”理解，而不是普通列表页：
  - 当前主卡完整展示
  - 下一张卡只从底部露出一部分
  - 视觉关系必须是主卡压住下一张卡，而不是下一张卡盖住主卡
  - 下一张卡必须与主卡同构，只是处于未完全展开状态，禁止再做成另一种“缩略卡组件”
- 宠物列表页的封面区只负责视觉识别，不承担重复信息表达：
  - 不要在封面区重复写状态、品种、记录数等信息
  - 信息统一收束到卡片中下部
  - 不要保留解释“下一张宠物档案”之类的冗余提示文案
- 宠物列表页主卡底部必须克制：
  - 只保留必要的状态信息
  - 只保留一个明确主 CTA `查看档案`
  - 不保留语义不清的小胶囊、占位按钮或重复次级入口
- 登录页 `登录页 Brand Welcome Login`（`52ZbF`）必须使用“单主卡”结构：
  - 顶部独立欢迎卡已删除，不允许再恢复双卡结构
  - 登录标题放入主卡内部并居中
  - 不保留英文眉标题
  - 不保留头像主体中央的装饰符号或无意义胶囊
- 登录页的资料获取必须按微信小程序当前能力实现，不允许写成“进入页即静默获取微信头像昵称”：
  - 进入页面先回填本地缓存或后端已有资料
  - 头像通过 `chooseAvatar` 获取并回填
  - 昵称通过 `input type=\"nickname\"` 获取并回填
  - 点击登录时再统一提交保存
- 地址编辑页 `地址编辑页 Address Edit`（`MRUGD`）中的地图搜索框已确认属于正式结构：
  - 搜索框位于“完善收货资料 / 地图选点”下方、地图预览上方
  - 画布至少要体现搜索框本体
  - 如果当前任务不是专门设计联想结果层，则不强制在 `.pen` 中同时展开下拉联想列表
- 页面文案必须做减法：
  - 如果层级、遮挡关系、图标或交互已经能表达含义，就不要再用解释性文案重复说明
  - 对高级感有害的典型冗余文案包括：下一步解释文案、重复状态描述、装饰性英文小标题、功能性弱提示

## 6. 统一实现流程

后续页面开发统一使用：

1. 读取 `.pen` 目标画布
2. 提取 page schema
3. 按 schema 实现 WXML / WXSS / JS
4. 用截图校验对照 `.pen`

其中针对小程序平台额外补充：

- `.pen` 中的 `status bar` 节点默认视为系统参考层，不进入页面 WXML
- 页面实现默认从业务内容容器开始，而不是从时间、信号、电量等系统信息开始
- 需要标题与返回能力时，优先使用微信原生导航栏承接

禁止：

- 直接凭印象翻译设计
- 边写代码边自己改设计
- 没有截图校验就声称“已 1:1 还原”

## 7. 当前重点交易链路

当前重点画布顺序：

1. 结算页
2. 支付方式弹窗
3. 支付验证层
4. 支付成功反馈
5. 地址管理页 / 地址编辑页

关键要求：

- 支付方式是覆盖在结算页上的底部弹层，不是跳转独立页
- 支付验证与支付成功都应保留系统级反馈感，不做普通结果页
- 地址链路按“地址管理页选择 + 地址编辑页修正保存”理解，地图选点与粘贴识别都属于地址编辑页内部能力

### 7.1 交易链补充规则（2026-03-16）

本轮对话已进一步确认以下规则，后续实现与联调都必须继续遵守：

对应的本轮执行进展、已完成修正、测试结果与剩余待验项，统一记录在：

- `docs/plans/2026-03-15-page-design-gap-remediation-plan.md`

- 当前交易链相关 `.pen` 关键画板统一按以下节点核对：
  - 结算页：`0fx0p`
  - 支付方式弹窗：`Dff1y`
  - Face ID 验证层：`Ef00X`
  - 支付密码输入态：`1bX6Z`
  - 支付成功反馈：`MeD2O`
  - 地址编辑页：`MRUGD`
- 地址编辑页必须包含“搜索框 + 地图 + 粘贴识别 + 表单修正”这一整套组合能力，不能退化成只有地图或只有表单的普通地址页。
- 地址编辑页中的地图搜索框必须遵循高德搜索建议的限流保护：
  - 少于 2 个字不触发搜索建议
  - 输入时必须做防抖，不能在每次键入时直接请求
  - 建议项字段缺失时必须使用兜底文案，禁止页面出现 `undefined`
- 地址编辑页中，当用户手动修改了地区或详细地址后，保存时不能继续沿用旧坐标；必须重新 geocode，或者明确要求用户重新搜索/选点确认地址。
- 地图与粘贴识别都是“辅助输入”，最终保存结果始终以编辑页当前字段值为准，不能以历史识别结果或旧地图状态反向覆盖表单。
- 结算页地址卡中的联系人与手机号必须按 `.pen` 做紧凑同排表达，避免旧实现中上下信息过松、间距过大的问题。
- 支付验证文案必须保持单行系统提示风格，例如：
  - `微信支付 · Face ID 验证`
  - `微信支付 · 支付密码验证`
- 支付成功后必须进入 `.pen` 定义的系统反馈态，再自动回流订单详情或商城；不能停留在原弹层状态，也不能卡死在旧页面栈中。

## 8. 后端协同原则

- 后端契约要适配已确认设计
- 接口字段命名、状态枚举、兼容策略要为当前页面形态服务
- 旧接口只能作为迁移兼容，不应继续成为新页面的设计边界
- 商品详情当前统一以 `specGroups`、`highlights`、`storySections`、`usageNote` 作为结构化内容契约
- 购物车、订单确认、订单提交、订单快照必须保留 `specLabel/specLabels`，保证不同商品规格能被后台独立配置、独立成交、独立追溯
